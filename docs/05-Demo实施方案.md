## 一、Demo 定位与范围

### 1.1 一句话目标

**4-6 周内做出一个可玩的「蜀 vs 蜀」单局 PvE demo**——玩家用刘备打 AI 控制的刘备，跑通完整对局闭环。

### 1.2 范围与不做

| ✅ 做 | ❌ 不做 |
|---|---|
| 1 个阵营（蜀），玩家 vs AI | 多阵营 / 在线 PvP |
| ~17 张卡（蜀 9 + 中立 8） | 完整 40 张牌池 |
| 6 个核心关键词 | 全部 13 个（剩 7 个 demo 后再加）|
| 完整对局闭环（换牌→回合→胜负） | 卡组构筑界面（牌组写死） |
| 简单贪心 AI | 强 AI / MCTS |
| 1 个战场背景 | 多场景 |
| 静态立绘 + CSS 动画 | 全 3D / 复杂粒子 |

### 1.3 为什么是「蜀 vs 蜀」

双方共用一套 9 张卡 + 8 中立，AI 测试时不会出现"一方卡池没实装"的诡异情况，且能集中力量打磨这一套卡。M1 阶段再扩展到「蜀 vs 魏」「蜀 vs 吴」。

### 1.4 Demo 实装关键词清单（6 个）

| 关键词 | 三国名 | 实现复杂度 | 为什么 demo 必须有 |
|---|---|:---:|---|
| Taunt | 镇守 | 低 | 战术核心（攻击优先级） |
| Charge | 冲锋 | 低 | 制造惊喜（OTK 风险展示）|
| Rush | 突袭 | 低 | 与冲锋的对照，教玩家区分 |
| Divine Shield | 铁壁 | 低 | 视觉效果好（金光盾，动画好做）|
| Battlecry | 威名 | 中 | 80% 战吼卡片必须的 |
| Deathrattle | 遗志 | 中 | 战斗深度（让死亡有意义）|

> 剩余 7 个关键词（连斩、淬毒、饮血、困阵、伏兵、谋略、离间）在 demo 后 1-2 周补齐，进入 M0 完整版。

```
精致工笔重彩 × 现代游戏 CG × 史诗气场
```

- **真·三国无双系列** 的角色立绘（华丽精致、细节丰富）

## 三、技术栈

### 3.1 最终选型

```yaml
核心:
  - React 18 + TypeScript 5
  - Vite 5（构建/热更新）

状态管理:
  - Zustand（极简，比 Redux 小 100 倍）

样式:
  - Tailwind CSS（原子化，AI 写得很顺）
  - CSS Modules（卡牌动画局部隔离）

动画:
  - Framer Motion（声明式，复杂动画首选）
  - 简单动画用 CSS transition

数据:
  - JSON 卡牌定义文件（可热更新）
  - 卡牌效果引擎用 TypeScript 函数表

开发工具:
  - ESLint + Prettier
  - Vitest（单元测试关键逻辑）

部署:
  - 开发：本地 localhost
  - 内测：Vercel free tier（一键部署，免费）
  - 后期：Tauri 打包成 Steam app
```

### 3.2 选型理由

| 选型 | 理由 |
|---|---|
| React 18 | AI 训练语料最大、组件化天然契合卡牌 |
| TypeScript | 卡牌效果系统类型校验救命稻草 |
| Vite | 热更新 < 100ms，对比 Unity 的 5-30s 编译 |
| Zustand | API 极简，整个游戏状态一个 store 搞定 |
| Tailwind | Claude 生成 CSS 非常顺，无需切换上下文 |
| Framer Motion | 卡牌进出场、攻击挥击的复杂动画首选 |

### 3.3 不选用的方案

| 方案 | 不选用理由 |
|---|---|
| Unity 2D | 上次踩坑：AI 看不到 Game View、编译慢、MCP 中转开销 |
| Godot 4 | AI 训练语料少于 Web，调试经验少 |
| libGDX (Java) | UI 全部要手写 OpenGL 画，卡牌游戏不划算 |
| Phaser.js | 偏 Canvas 渲染，UI 重的卡牌游戏用 DOM 更省心 |
| Cocos Creator | 国内主流但 AI 协作不如 React 顺 |

## 五、核心数据结构

### 5.1 卡牌 JSON Schema

```typescript
interface CardData {
  id: string                  // "S01" 等唯一 ID
  name: string                // "关羽"
  cost: number                // 法力费用
  type: "minion" | "spell" | "weapon"
  faction: "shu" | "wei" | "wu" | "qun" | "neutral"
  rarity: "common" | "rare" | "epic" | "legendary"

  // 武将专属
  attack?: number
  health?: number
  tribe?: "infantry" | "cavalry" | "strategist"

  // 兵器专属
  durability?: number

  // 关键词数组（实现表里有对应函数）
  keywords?: string[]         // ["taunt", "charge"]

  // 效果列表（按触发时机）
  effects?: {
    trigger: "battlecry" | "deathrattle" | "endTurn" | "onCast"
    action: string            // 函数名，如 "dealDamage"
    params?: any              // 参数
  }[]

  // 视觉
  portrait: string            // 立绘文件名
  description: string         // 卡牌文本
  flavor?: string             // 风味文字（小字）
}
```

## 六、关键流程实现

### 6.1 出牌流程（拖拽）

```
1. 玩家在手牌区按住卡牌
2. CSS 高亮可放置区域（战场 / 目标）
3. 释放：
   - 落在战场 → 召唤武将（检查法力 + 战场上限）
   - 落在目标 → 计策施法（检查目标合法性）
4. 触发威名（如有）
5. 状态更新 → React 重新渲染
6. 动画反馈：卡牌从手牌飞到目标位置
```

### 6.2 攻击流程

```
1. 玩家点击战场上自己的武将（高亮可攻击目标）
2. 检查嘲讽：对方场上若有镇守，只能攻击镇守
3. 玩家点击目标 → 触发攻击
4. 同时结算双方伤害（炉石原版规则）
5. 检查死亡：生命值 ≤ 0 → 进入坟场，触发遗志
6. 动画：攻击者向目标冲撞，受伤抖动，死亡淡出
```

### 6.3 回合切换

```
回合结束按钮 →
  1. 触发"回合结束时"效果
  2. 移除临时 buff
  3. 切换 activePlayer
  4. 触发"回合开始时"效果
  5. activePlayer 抽 1 张牌（先手第 1 回合除外）
  6. 如果是 AI 回合 → 调用 greedyAI.takeTurn()
```

### 6.4 AI 决策（贪心）

```typescript
// 极简 AI：每回合按以下优先级行动
function aiTurn(state) {
  // 1. 用完所有可用法力打牌（高费优先）
  while (canPlayCard()) {
    playHighestCostCard()
  }

  // 2. 所有可攻击单位攻击
  // 攻击优先级：
  //   - 优先解掉对方场上威胁（攻击 ≥ 4 的随从）
  //   - 其次平交对方随从
  //   - 最后无视嘲讽前提下打脸
  for (const minion of myBoard) {
    if (canAttack(minion)) {
      attackBestTarget(minion)
    }
  }

  // 3. 用英雄技能（如果还有 2 法力）
  if (mana >= 2 && !heroPowerUsed) {
    useHeroPower()
  }

  // 4. 结束回合
  endTurn()
}
```

这个 AI 不会赢专业玩家，但能跑出有趣的对局供测试。M1 再加深度。

| **W5** | AI + 主菜单 + 视觉打磨 | 贪心 AI；主菜单；CSS 动画（卡牌进出场、攻击挥击、死亡淡出）；音效 |
| **W6** | bug 修 + 平衡测试 | AI vs AI 自动对战 100 局收集胜率；修关键 bug；录制 demo 视频 |

> **关键风险点**：W3 的关键词系统是最复杂的。如果 W3 拖到 W4 也属正常——MVP 优先保证完整对局，关键词可以先实装 4 个（镇守/威名/遗志/冲锋），其他延后。

## 九、风险与降级方案

### 9.1 三大风险

| 风险 | 概率 | 影响 | 降级方案 |
|---|:---:|---|---|
| **美术资源延期** | 高 | demo 没立绘只能用占位图 | 用 AI 生成立绘临时顶替；最差用纯文字卡牌 |
| **关键词效果系统设计崩溃** | 中 | 卡牌效果互相影响出 bug | 先做 4 个简单关键词，复杂的（威名/遗志）先实现单一形态 |
| **AI 行为太蠢/太强** | 低 | 测试对局没意思 | 调贪心权重；加随机扰动 |

### 9.2 时间降级

如果 6 周不够：

```
最低交付：
  - 蜀 vs 蜀 单局对战
  - 4 个关键词（镇守 / 威名 / 遗志 / 冲锋）
  - 12 张卡（蜀 9 + 中立 3）
  - 简陋 AI（随机出牌 + 见到就攻击）

砍掉：
  - 视觉打磨
  - 主菜单（直接进对局）
  - 音效音乐
  - 动画细节
```

确保**6 周内一定有可玩 demo**——哪怕粗糙也比"还差一点完美"好。

### 9.3 美术兜底

完整美术清单见 [06-美术资源清单.md](06-美术资源清单.md)。如果美术全部延期：

- 卡牌底框 → 纯 CSS 绘制（边框 + 渐变 + 阴影），可临时使用
- 武将立绘 → AI 生成（Midjourney / SD）+ 简单后期
- UI 图标 → 用 Lucide / Heroicons 等开源图标库临时替代
- 字体 → 思源宋体免费足够
- 音效 → freesound.org CC0 资源

**先验证玩法，再砸美术**。
