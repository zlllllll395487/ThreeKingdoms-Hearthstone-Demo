## 1. 设计哲学与硬规则

### 1.1 设计哲学

- **可玩性优先于真实感**：数值靠拢炉石 vanilla 公式（1 法力 ≈ 2 stats），保证策略空间
- **阵营自洽**：每个阵营有独有机制驱动整套牌组节奏，不只是 flavor
- **demo 体量克制**：单局 12-15 分钟、4 种对局组合、55 张卡足够 20 局体验差异化
- **数据驱动美术**：先 lock 玩法数值，确认后才出立绘 + cardvisual

### 1.2 阵营定位

| 阵营 | 定位 | 风格 |
|---|---|---|
| **蜀** | 实体铺场与滚雪球引擎 | 极端的场面压制 / 实体衍生 / 战阵规模反哺核心单位 |
| **吴** | 将谋联动与资源流转引擎 | 法术控制流 / 前期防御化解 / 中后期资源爆发 |

### 1.3 设计隔离硬规则（绝对不可破）

❌ **蜀禁用**：
- 高效群体伤害法术（万箭/火烧类）
- 直接抽牌法术
- 法术召唤 token（铺场只能靠实体交互）

❌ **吴禁用**：
- 铺满全场的高 stat 近战兵
- 无脑直接抽牌（所有过牌必须依附联动）
- token 召唤铺场

✅ **JSON 数据预留字段**：
- `anchorTag` / `anchorRequirement` / `linkedEffects` 给锚点系统
- `comboFlagSet` / `comboFlagRequirement` / `comboLinkedEffects` 给卡-卡联动
- UI 必须实时轮询场上锚点状态，高亮可联动手牌

## 3. 蜀阵营完整设计

### 3.1 核心机制 · 3 大支柱

#### A · 实体铺场引擎（完全剥离法术铺场）

铺场只来自 4 种实体交互：
1. **兵器主动攻击触发**：主公用特定武器攻击后召唤伴生（雌雄双股剑）
2. **武将战吼召唤**：关平 / 严颜 等战吼直接铺兵
3. **武将亡语召唤**：破虏校尉（S09） / 陷阵老卒（S13） / 校尉（S14） 死亡时铺兵
4. **武将受击召唤**：黄月英 受到伤害未死时铺兵（每张每局 1 次）

#### B · 个体级额外攻击（禁止全局机制）

- **五虎合击** 5 费法术：选**一个**友方武将本回合再攻击 1 次
- 绝对禁止全局 windfury / 全场额外回合（避免破坏系统）

#### C · 滚雪球公式（动态属性叠加）

- 关羽 / 张飞 / 赵云 入场战吼：数己方半场**其他**友方武将数 N → 本武将 +N/+N
- 一次性结算，不持续叠加
- 上限：满场 7 友方 - 1 自己 = 6N → 例：关羽满场 = 10/10

## 4. 吴阵营完整设计

### 4.1 核心机制 · 3 大支柱

#### A · 锚点联动（武将 → 谋略）

3 锚点武将定义谋略卡的「质变触发条件」：

| 锚点武将 | 标签 | 在场时激活 |
|---|---|---|
| **周瑜** | `anchor_fire` | 所有「火」谋略联动效果 |
| **鲁肃** | `anchor_draw` | 所有「抽牌」谋略联动效果 |
| **大乔** | `anchor_heal` | 所有「治疗/防御」谋略联动效果 |

**判定时机**：玩家打出谋略卡的瞬间，扫描己方场上是否存在 anchorRequirement 匹配的武将。
**UI 强引导**：手牌中可联动的谋略卡**实时高亮**（金光描边+微抖动）。

#### B · 卡-卡联动（谋略 → 谋略）

2 组卡-卡组合，通过本回合内的 flag 触发：

| 前置卡 | 后置卡 | 联动效果 |
|---|---|---|
| **火油**（设 `combo_fire` flag）| **赤焰焚营** | 单体伤害 3 → 6 |
| **反间计**（设 `combo_betrayal` flag）| **美人计** | 夺取 ≤3 费 → ≤5 费 + 归还 2 费 |

**判定时机**：打前置卡时设置 flag（仅本回合内有效），打后置卡时检查 flag 触发联动效果。
**回合结束**：清空所有 comboFlagsThisTurn。

#### C · 法力返还（破上限）

- 联动结算后**当回合**法力 +N（可继续出牌）
- 不影响下回合上限
- 例：火烧赤壁 7 费 + 周瑜锚点 → 6 伤 AoE + 当回合归还 3 费 → 可继续打 3 费的牌
- 设计意图：禁用「-费」机制（费用减免破坏数值），改用「返费」让玩家做更复杂的回合规划

## 7. 联动系统

### 7.1 武将-谋略联动（锚点）

#### 数据结构

```json
// 锚点武将（W01 周瑜示例）
{
  "id": "W01",
  "name": "周瑜",
  "anchorTag": "anchor_fire",
  ...
}

// 谋略卡（W09 火烧赤壁示例）
{
  "id": "W09",
  "name": "火烧赤壁",
  "cost": 7,
  "tags": ["fire"],
  "anchorRequirement": "anchor_fire",
  "effects": [
    { "trigger": "onCast", "action": "dealDamageAll", "params": { "side": "enemy", "amount": 4 } }
  ],
  "linkedEffects": [
    { "trigger": "onCast", "action": "dealDamageAll", "params": { "side": "enemy", "amount": 2 } },
    { "trigger": "onCast", "action": "refundMana", "params": { "amount": 3 } }
  ]
}
```

#### 触发逻辑

```
playCard(card):
  if card.anchorRequirement and player.board has minion with matching anchorTag:
    执行 effects (基础) + linkedEffects (联动)
  else:
    执行 effects (基础)
```

#### UI 高亮

```
对手牌每张卡：
  if card.anchorRequirement and player.board has minion with matching anchorTag:
    渲染 data-linkable='true' → CSS 金光描边动效
```

### 7.2 谋略-谋略联动（卡-卡）

#### 数据结构

```json
// W18 火油（前置卡）
{
  "id": "W18",
  "name": "火油",
  "comboFlagSet": "combo_fire",
  "effects": [
    { "trigger": "onCast", "action": "cannotAttackThisTurn", "params": { "target": "enemyMinion" } }
  ]
}

// W19 赤焰焚营（后置卡）
{
  "id": "W19",
  "name": "赤焰焚营",
  "cost": 3,
  "comboFlagRequirement": "combo_fire",
  "effects": [
    { "trigger": "onCast", "action": "dealDamage", "params": { "amount": 3, "targetSelector": "enemyAny" } }
  ],
  "comboLinkedEffects": [
    { "trigger": "onCast", "action": "dealDamage", "params": { "amount": 3, "targetSelector": "enemyAny" } }
  ]
}
```

#### 触发逻辑

```
playCard(card):
  if card.comboFlagRequirement in player.comboFlagsThisTurn:
    执行 effects (基础) + comboLinkedEffects (联动)
  else:
    执行 effects (基础)

  if card.comboFlagSet:
    player.comboFlagsThisTurn.add(card.comboFlagSet)

endTurn():
  player.comboFlagsThisTurn.clear()  // 仅本回合内有效
```

#### UI 高亮

```
对手牌每张卡：
  if card.comboFlagRequirement in player.comboFlagsThisTurn:
    渲染 data-combolinked='true' → CSS 蓝光描边动效
```

### 7.3 法力返还机制

#### 行为

- `refundMana` action：`player.mana.current += amount`
- 不影响 `player.mana.max`（下回合上限不变）
- 当回合内可继续出牌

#### 示例（火烧赤壁 + 周瑜）

```
当前 mana: 7/7
打火烧赤壁 7 费 → mana 变为 0/7
检测周瑜在场 → linkedEffects 触发：
  + dealDamageAll +2 (全敌方 6 伤总)
  + refundMana 3
mana 变为 3/7
玩家可继续打 ≤ 3 费的牌
```

### 7.4 UI 高亮规范

| 状态 | CSS 视觉 | 触发时机 |
|---|---|---|
| `data-linkable='true'` | 金光描边 + 微抖动 + 边框波纹 | 锚点武将在场 → 对应谋略卡 |
| `data-combolinked='true'` | 蓝光描边 + 闪烁 | 本回合 combo flag 已设 → 对应后置卡 |
| 两者叠加 | 双色描边（金外蓝内）| 同时满足时 |

**实时轮询**：每次 state 变化触发 React 重渲染检测，自动更新所有手牌的高亮状态。

**结果**：吴 T6-T7 清场及时则胜。**预期胜率 50/50 ± 5%**。

吴 T5-T6 锚点 setup 提前一拍 → 吴胜率略高 +3-5%（先手优势）。

### 9.4 关键风险点与缓冲

| 风险 | 缓冲机制 |
|---|---|
| 关羽 +N/+N 失控（v5.3 削弱后）| base 4/4 限制，满场 N=6，上限 10/10 突袭（v5.3 去武器附赠后，face 爆发空间被切除） |
| 吴清场过强 | 火烧赤壁 7 费 + 锚点 setup（需 5 费周瑜 + T4-T6 投入）门槛高；W10 火攻连环 v5.3 删，吴清场只剩 W09 一张 |
| 周郎顾曲 2 费过强 | v5.4 联动 = 冻结 + 攻 -1 本回合；联动需 5 费周瑜 setup；纯临时软控，无永久 debuff 体感问题；mana value 回归 2.5-3.0 健康区间 |
| 平局规则（PvE 法则） | 同回合双方 HP ≤ 0 时**强制判玩家胜利**（playerDead && aiDead → winner = 'player'），播 Victory 动画。理据：PvE 体验的最高法则是"让玩家爽"，平局结算屏会带来挫败感 |
| 缓兵之计相邻位机制 | 仅锁定 2 个敌方武将的攻击节奏；不造成伤害；引擎需新增 board adjacency 字段 |
| 卡-卡联动滥用 | 每对仅 2 张卡，flag 仅本回合内有效，强制连续 2 回合规划 |
| 法力返还破天际 | 单次最多归还 3 费（火烧赤壁），且 7 费基础门槛 |
| 受击触发死循环 | 黄月英每张每局 1 次（onceUsedKey 限制） |

### 9.5 自动 playtest 方案

**脚本**：`scripts/simulate.ts`
- 4 种对位各跑 25 局 → 共 100 局
- 输出胜率 + 平均回合长度 + 关键卡使用频率

**调整规则**：
- 胜率偏离 ±5% → 调数值
- 蜀过强 → 优先调关羽 base 4/4 → 3/3 或 +N/+N → +N/+0（v5.3 去武器后蜀爆发空间已削减）
- 吴过强 → 优先调火烧赤壁联动归还 3 费 → 2 费 / 周郎顾曲联动抽 1 张 → 抽 0 张
- 调武将 stat 1 点 ≈ 调胜率 2-3%
- 严禁同时多卡同方向调整（避免连锁）
