## 1. 作品概述

《三国炉石》是一款以三国为题材、炉石传说为玩法骨架的卡牌对战游戏 Web 端 Demo。

| 项 | 内容 |
|:--|:--|
| 类型 | 回合制集换式卡牌对战（CCG） |
| 平台 | Web（桌面与移动端浏览器自适应） |
| 技术栈 | React 19 · TypeScript 6 · Vite 8 · Zustand 5 · CSS Modules · Framer Motion |
| 对战形态 | 单人对战 AI（三档难度）· 在线真人对战（房间码约战） |
| 卡牌规模 | 66 张（蜀 23 / 吴 29 / 中立 10 / 衍生物 4） |
| 平衡支撑 | 内置 AI 对战模拟框架，1000 局约 1.5 秒，数据驱动数值决策 |
| 在线试玩 | https://zlllllll395487.github.io/ThreeKingdoms-Hearthstone-Demo/ |

核心体验定位：**上手零门槛**（炉石玩家可即时理解），**策略有新意**（三国阵营机制提供差异化深度）。

## 2. 设计哲学与硬规则

### 2.1 设计哲学

- **可玩性优先于真实感**：基础数值靠拢炉石 vanilla 公式（1 法力约等于 2 点属性），保证策略空间稳定。
- **阵营自洽**：每个阵营由独有机制驱动整套牌组节奏，而非仅作题材包装。
- **体量克制**：Demo 聚焦蜀、吴两阵营与有限卡池，确保单局体验完整且差异化。
- **数据驱动**：先锁定玩法数值，经模拟验证后再推进美术与表现层。

### 2.2 阵营定位

| 阵营 | 主公 | 定位 | 风格 |
|:--|:--|:--|:--|
| **蜀** | 刘备 | 实体铺场与滚雪球引擎 | 场面压制、实体衍生、战阵规模反哺核心单位 |
| **吴** | 孙权 | 将谋联动与资源流转引擎 | 法术控制、前期防御化解、中后期资源爆发 |

### 2.3 设计隔离硬规则（不可破）

阵营差异化依赖以下隔离规则，确保两套牌组节奏不趋同：

**蜀禁用**：
- 高效群体伤害法术（铺场只能靠实体单位交互）
- 直接抽牌法术
- 由法术召唤衍生物铺场

**吴禁用**：
- 铺满全场的高属性近战单位
- 不依附联动的无条件抽牌（所有过牌须挂靠联动机制）
- 由法术召唤衍生物铺场

## 3. 基础规则

| 系统 | 规则 |
|:--|:--|
| 主公生命 | 双方主公 30 点生命值，先降至 0 者败北 |
| 法力 | 每回合法力上限 +1（封顶 10），回合开始时回满 |
| 起手 | 先手 3 张、后手 4 张；内置起手保证机制（1、2、3 费各保证 1 张可玩卡） |
| 抽牌 | 第 1–5 回合每回合抽 1 张，第 6 回合起每回合抽 2 张，缓解后期断流 |
| 疲劳 | 牌库抽空后继续抽牌触发递增疲劳伤害 |
| 战场 | 每方最多 7 个武将同时在场 |
| 回合结构 | 回合开始（抽牌、回能）→ 主要阶段（出牌、攻击）→ 回合结束 |

> 部分炉石机制在 Demo 中尚未实装或作简化处理，详见 [第 10 节 · 未来规划](#10-未来规划)。

## 4. 阵营设计

### 4.1 蜀 · 实体铺场与滚雪球

蜀阵营以实体武将铺场为核心，通过衍生物（民兵等 token）扩大场面规模，并由战阵规模反哺核心单位的属性成长。其压制力来自场面雪球，而非法术爆发。

代表性机制：

- **衍生物铺场**：部分武将与兵器登场或攻击时召唤民兵 token，持续施加场面压力。
- **战阵反哺**：核心单位的属性随友方场面规模或特定条件成长。
- **兵器体系**：兵器仅可由主公装备（武将不可持兵器），为主公提供主动进攻手段。

### 4.2 吴 · 将谋联动与资源流转

吴阵营以法术（计策）与武将的联动为核心，前期以防御性法术化解攻势，中后期通过资源流转完成爆发。其过牌与爆发均依附于联动机制，符合阵营隔离规则。

吴阵营拥有两套专属联动系统：

**锚点系统（anchor）** —— 场上存在带特定锚点标签的友方武将时，对应法术触发额外联动效果：

| 锚点 | 主题 |
|:--|:--|
| `anchor_fire` | 火攻强化 |
| `anchor_draw` | 过牌增益 |
| `anchor_heal` | 治疗增益 |
| `anchor_ramp` | 法力加速 |

**卡-卡联动（combo）** —— 本回合已施放特定标记卡后，后续触发卡获得连击强化：

| 联动标记 | 主题 |
|:--|:--|
| `combo_fire` | 火油 → 火攻连锁 |
| `combo_betrayal` | 反间计 → 控场连锁 |

UI 在战斗中实时高亮可联动的手牌，引导玩家组织联动链。

## 5. 卡牌总览

当前卡池共 66 张，构成如下：

| 类别 | 张数 | 说明 |
|:--|:-:|:--|
| 蜀 | 23 | 武将、计策、兵器 |
| 吴 | 29 | 武将、计策、兵器 |
| 中立 | 10 | 双方通用 |
| 衍生物（token） | 4 | 由卡牌效果召唤，不可直接组入牌库 |

卡牌类型分为**武将**（minion）、**计策**（spell）、**兵器**（weapon）三类。

> 单卡的精确属性、费用与效果以代码数据为唯一来源：`game/src/data/cards/{shu,wu,neutral,tokens}.json`；
> 人工可读的逐卡验收清单见 [docs/12-卡牌验收清单-v5.2.md](12-卡牌验收清单-v5.2.md)。
> 本文档不复制逐卡数值，以避免随平衡迭代而失效。
>
> 注：魏、群两阵营在图鉴中已预留分页，但当前**无卡牌数据**，不在 Demo 范围内。

## 7. 数值平衡体系

为支撑数值与机制层面的平衡决策，项目内置一套 **AI 对战模拟框架**（`game/scripts/sim/`），将调整建立在统计数据之上而非主观经验。

### 7.1 方法论

1. **建立基线**：执行 1000 局模拟，确立未经主观干预的数据基准。
2. **AI 决策审计**：通过单局 trace 逐回合审查 AI 决策，区分「AI 决策缺陷」与「卡牌结构性弱势」。
3. **卡牌微调**：优先增强 Bottom 弱卡、削弱 Top 强卡，避免大幅结构性改动。
4. **验证迭代**：以新一轮 1000 局数据验证效果，目标是阵营差进入 ±10% 可接受区间。
5. **避免过度修正**：出现过度修正时优先回退，而非引入新机制补救。

### 7.2 迭代收敛史

| 阶段 | 蜀总胜率 | 吴总胜率 | 阵营差 | 备注 |
|:--|:-:|:-:|:-:|:--|
| baseline | 99.3% | 34.0% | 65.3% | 严重失衡，吴方系统性弱势 |
| iter2 | 99.2% | 34.1% | 65.1% | 机制改造，体验指标恢复健康 |
| iter3 | 90.3% | 43.1% | 47.2% | 补充吴卡，起手卡死归零 |
| iter4 | 85.1% | 48.3% | 36.8% | 修复 5 处 AI 决策缺陷 |
| iter5 | 62.1% | 71.2% | 9.1% | 11 张卡数值调整，进入可接受区间 |
| **iter6.1** | **69.5%** | **63.9%** | **5.6%** | **全部核心指标进入健康区间** |

**当前结论**：经 6 轮迭代，阵营差由 65.3% 收敛至 5.6%，跨阵营双向胜率均落入 ±10% 区间，起手卡死率回归 0%。本阶段为 v5.6 数值平衡的验收版本。详细迭代记录见 [README.md](../README.md) §22 与 [docs/sim-reports/](sim-reports/)。

> 运行模拟与生成胜率报告的工具，已封装为双击启动器（见 [第 9 节](#9-已实现功能清单)）。

## 8. 在线对战

在线对战采用**权威服务器**模型，服务器复用与前端同一份游戏引擎作为裁判，从根本上规避「服务端与客户端规则不一致」这一多人游戏常见缺陷。架构决策详见 [DECISIONS.md](DECISIONS.md) D-012。

### 8.1 核心架构

- **权威服务器**：服务器运行唯一的游戏引擎，客户端仅发送动作意图（出牌 / 攻击 / 结束回合），由服务器校验并执行。
- **个性化脱敏 + 视角翻转**：服务器为每个客户端单独下发状态，使接收者恒在「我方」视角、对手手牌脱敏为牌背。前端战斗界面因此无需为在线模式改动视角逻辑。
- **前端瘦客户端**：在线模式下前端不持有权威引擎，仅以服务器状态构造只读「影子引擎」用于界面判定（是否可出牌、是否需要目标等）。

### 8.2 当前能力

- 房间码约战：创建房间得房间码，对手凭码加入，房主开始对战。
- 完整对战同步：出牌、攻击、结束回合经服务器裁决后同步双方，分出胜负。
- 对局收尾：对手中途离开有提示并可退出，对局结束后连接与状态正确清理。

### 8.3 部署形态

当前以本地后端配合内网穿透（ngrok 等）作为 MVP 验证形态；公开匹配与云端常驻部署见 [第 10 节 · 未来规划](#10-未来规划)。
